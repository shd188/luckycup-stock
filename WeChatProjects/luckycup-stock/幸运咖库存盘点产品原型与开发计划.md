# 幸运咖库存盘点小程序 - 产品原型与开发计划（V1）

> 本文档用于沉淀当前版本的产品方案与技术方案，为后续版本迭代提供依据。

---

## 一、产品目标与定位

- **产品目标**：为幸运咖门店提供一款「简单可靠的库存盘点工具」，按总部统一的物料结构进行盘点，沉淀结构化数据，为后续数据分析与智能化能力打基础。
- **定位**：不是一次性表单，而是「幸运咖门店数据资产系统的底层基础设施」中的第一个环节。

---

## 二、角色与使用场景

- **使用角色**
  - 门店店长
  - 门店员工
- **不使用本小程序的角色**
  - 总部运营 / 总部管理员（物料结构维护通过脚本或其他后台完成，不在本小程序内）

---

## 三、核心业务规则（确认版）

1. **物料分类结构**
   - 固定为两级结构：分类（category）→ 物料（material）。
   - 不会增加第三级结构。
   - 所有门店共用同一套分类和物料结构。
   - 物料名称全局唯一。

2. **物料生命周期**
   - 物料会新增。
   - 物料会删除（逻辑删除，`is_active=0`）。
   - 物料顺序会调整（通过 `sort_id` 控制）。
   - 单位不统一（个 / 瓶 / 条 / 包 / 盒 等）。
   - 小程序**不负责 Excel 导入**，物料结构通过脚本或外部后台维护，小程序只负责读取和展示。

3. **门店与用户**
   - 一个用户（微信 `openid`）可以管理多个门店。
   - 一个门店可以被多个用户「加入」并管理。
   - 门店以 `store_code` 为全局唯一标识，一个门店只能被创建一次。

4. **库存填写与校验**
   - 填写页按「分类 → 物料」顺序展示。
   - 每个物料对应一个库存输入框。
   - 允许数量为 `0`。
   - 禁止：
     - `null`
     - `""`（空字符串）
     - 未填写
   - 正确校验逻辑示意：

```js
if (value === '' || value === null) {
  return error
}
```

5. **盘点规则**
   - 一个门店一天可以盘点多次（多次盘点都是独立快照）。
   - 提交前必须保证**所有活跃物料都有填写**。
   - 提交前弹出二次确认：「库存提交后不可修改，是否确认提交？」。
   - 提交后当前盘点不可修改。

6. **分享与公开查看**
   - 通过 `inventory_id` 作为分享参数。
   - 打开分享页不要求登录/授权（只读查看）。
   - 分享页展示：
     - 门店编号
     - 门店名称
     - 录入完成时间
     - 盘点人昵称
     - 所有物料库存数量
   - 排序规则：
     - 先按 `category_sort_id`
     - 再按 `material_sort_id`
   - 当某物料当前 `is_active=0` 时：
     - 填写页：不展示该物料。
     - 分享页：如果历史快照中有该物料，则仍然展示，并在物料名称右侧标记「已下架」。

## 四、数据库设计（V1 修订版）

### 1. categories（分类表）

```js
{
  _id,
  name,
  sort_id,
  is_active,
  created_at
}
```

### 2. materials（物料表）

```js
{
  _id,
  name,         // 物料名称，全局唯一
  category_id,
  unit,         // 单位（个 / 瓶 / 包 / ...）
  sort_id,
  is_active,
  created_at
}
```

排序规则：

- 先按 `category.sort_id` 升序
- 再按 `material.sort_id` 升序

### 3. users（用户表）

```js
{
  _id,
  openid,
  nickname,     // 微信昵称（通过微信授权获取）
  avatar,       // 微信头像（可选）
  created_at
}
```

说明：

- 为符合微信审核机制，盘点人昵称**建议直接使用微信昵称**。
- 获取方式：在需要展示盘点人昵称时，通过微信推荐的用户信息接口（如 `wx.getUserProfile` 或平台当前推荐方式）发起一次授权，说明用途为「用于展示盘点人昵称」。

### 4. stores（门店表）

> V1 对原始设计做了轻微调整：门店与用户的多对多关系通过中间表维护。

```js
{
  _id,
  store_code,   // 门店编号，全局唯一
  store_name,
  created_at
}
```

说明：

- 一个 `store_code` 在此表中只会出现一次。

### 5. user_stores（用户-门店关系表）

```js
{
  _id,
  user_id,
  store_id,
  role,         // 角色（如 'owner' | 'staff'），V1 可选填，默认 'staff'
  created_at
}
```

说明：

- 支持：
  - 一个用户管理多个门店。
  - 一个门店被多个用户加入。

### 6. inventories（盘点主表）

```js
{
  _id,
  store_id,
  user_id,       // 发起本次盘点的用户
  finished_at,   // 完成时间
  created_at
}
```

### 7. inventory_items（盘点明细表）

每个物料一条记录（快照存储）：

```js
{
  _id,
  inventory_id,

  category_name_snapshot,
  category_sort_id,

  material_name_snapshot,
  material_sort_id,
  unit_snapshot,

  quantity
}
```

说明：

- 不保存 `material_id`。
- 每次盘点为独立数据快照。
- 历史数据不受结构变化影响。

## 五、Excel 模板（物料导入，仅供后台/脚本使用）

当前版本小程序**不实现 Excel 导入界面**，但保留结构说明：

| category_name | category_sort_id | material_name | material_sort_id | unit | is_active |
| ------------- | ---------------- | ------------- | ---------------- | ---- | --------- |
| 分类A         | 1                | 产品1         | 1                | 个   | 1         |
| 分类A         | 1                | 产品2         | 2                | 瓶   | 1         |
| 分类B         | 2                | 产品3         | 1                | 包   | 1         |
| 分类B         | 2                | 产品4         | 2                | 盒   | 1         |

规则：

- `material_name` 不可重复。
- 删除通过 `is_active=0` 控制。
- 排序通过 `sort_id` 控制。

## 六、用户流程与页面原型（概要）

### 1. 门店列表页（首页）

- 展示当前用户已加入的门店列表（从 `user_stores` + `stores` 查询）。
- 支持新增/加入门店：
  - 用户输入 `store_code` 和 `store_name`。
  - 云端检查：
    - 若 `store_code` 不存在：
      - 创建 `stores` 记录。
      - 创建 `user_stores` 记录，将当前用户与该门店关联。
    - 若 `store_code` 已存在：
      - 返回数据库中的 `store_name`。
      - 前端弹出确认：
        - 文案示例：  
          `系统已存在门店：{store_name}（编号 {store_code}），如果这是你的门店，可选择加入后开始盘点。`
        - 按钮：`取消` / `加入该门店`
      - 用户确认后：
        - 创建 `user_stores` 记录，将当前用户加入该门店。
- 点击某个门店卡片 → 进入「库存盘点填写页」。

### 2. 库存盘点填写页

- 顶部展示当前门店名称和编号。
- 中间按「分类 → 物料」展示所有**活跃物料**，每行一个数量输入框。
- 底部提供「提交盘点」按钮。
- 提交时进行**全量非空校验**，允许 `0`，禁止空值。
- 二次确认弹窗后创建 `inventories` + `inventory_items` 记录。

### 3. 盘点提交成功页

- 展示：
  - 门店名称和编号。
  - 完成时间。
  - 盘点人昵称（来自 `users.nickname`，即微信昵称）。
- 提示「本次盘点已锁定，不可修改」以及「可通过右上角菜单转发查看」。

### 4. 盘点结果分享页

- 入参：`inventory_id`。
- 无需登录/授权即可打开，仅展示数据。
- 展示：
  - 门店编号、门店名称。
  - 完成时间。
  - 盘点人昵称。
  - 分类 + 物料列表（来自 `inventory_items` 快照）。
  - 当前已下架物料标记「已下架」。

## 七、技术架构与实现要点（简要）

1. **技术栈**
   - 微信小程序（前端）。
   - 微信云开发：云函数 + 云数据库。

2. **用户昵称（微信审核相关）**
   - 只在需要展示「盘点人」时请求一次用户信息授权。
   - 授权说明文案应明确：「用于展示盘点人昵称」。
   - 授权后仅保存必要字段（昵称、头像），并存入 `users` 表。

3. **门店加入逻辑**
   - 后端负责根据 `store_code` 判断门店是否存在。
   - 若存在，通过 `user_stores` 维护用户与门店的关联关系。
   - 确保一个 `store_code` 在 `stores` 中唯一，多个用户可以通过 `user_stores` 加入同一门店。

4. **数据快照**
   - 盘点明细全部通过快照方式存储，避免后续物料结构变化影响历史数据。

## 八、页面结构与代码规划（概要）

小程序主要页面规划（路径与用途）：

- `page/store-list/index`：门店列表页（首页）。
- `page/store-form/index`：新增/加入门店页。
- `page/inventory-edit/index`：库存盘点填写页。
- `page/inventory-success/index`：盘点提交成功页。
- `page/inventory-result/index`：盘点结果分享页。

后续开发会在上述路径下实现对应的 `wxml / wxss / js / json` 文件，并在 `app.json` 中进行注册。
